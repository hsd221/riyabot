# 消息链路改造参考计划

## 背景

上游新版 MaiBot 的核心变化不是单点修复，而是把消息链路拆成更清晰的边界：

- 入站消息快速接收，耗时媒体处理后置。
- 出站发送集中到服务层。
- 回复触发逻辑从聊天循环中抽出。
- 插件可在关键节点通过 hook 观察或改写流程。
- 消息内容逐步从 `Seg` 字符串分支转向标准组件。

本项目不建议一次性跟完整 Maisaka 架构。更稳妥的做法是按收益和风险拆成小步迁移。

## 建议顺序

1. `inbound-lightweight-step1.md`
   - 给入站消息处理加轻量模式。
   - 先避免图片、表情、语音同步堵住主链路。

2. `inbound-media-background-step2.md`
   - 在轻量模式基础上增加后台媒体识别和缓存回填。
   - 让后续 prompt 构建能拿到已完成的图片/表情/语音描述。

3. `send-service-step3.md`
   - 把发送构造、typing、引用回复、发送前后事件、写库收敛成统一发送服务。
   - 降低后续改平台适配、WebUI 虚拟群、消息回填的成本。

4. `reply-turn-scheduler-step4.md`
   - 把群聊/私聊的回复触发判断从循环体抽成调度器。
   - 统一处理 @ 必回、频率阈值、私聊 wait、静默模式等规则。

5. `message-hooks-step5.md`
   - 补齐入站、命令、规划、发送的关键 hook 点。
   - 给插件和本地自定义逻辑留稳定扩展口。

6. `message-components-step6.md`
   - 逐步引入内部消息组件模型。
   - 减少到处判断 `Seg.type` 的重复逻辑。

## 不建议优先做的内容

以下内容可以作为长期方向，但不建议作为当前第一批改动：

- 完整移植 Maisaka runtime。
- 完整替换为上游 `plugin_runtime` 子进程插件系统。
- 完整接入 `platform_io` 多驱动路由层。
- 完整替换本地 `src/memory` 为 A_memorix。

这些改动牵动消息模型、数据库、插件、WebUI 和配置体系，适合作为以新版上游为基底重做分支时再处理。
